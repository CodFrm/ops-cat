## 2. 结论分类

### 2.1 总分类

本轮正式分类为：

- `已确认的首开定位 / 首帧可见性缺陷`

明确不再维持上一轮口径：

- `未确认 bug，但存在可见性 / 预期问题`

明确不收口为：

- `独立新增 UX 要求`

### 2.2 当前冲突高亮

结论：

1. “当前冲突高亮应当在首帧可见”仍然属于当前 PR 既定行为。
2. 但 owner 最新现场观察已经把问题从“是否只是感知不明显”提升为“第一次打开时看不到冲突点，必须手动触发导航后才看到”。
3. 因此本轮 requirement truth 必须按缺陷收口，而不是继续停留在未证实口径。

这意味着：

1. 问题不再只是 reviewer-facing 文案不充分。
2. 问题已进入用户首开体验缺陷边界。
3. 后续 SA / DEV 应按 bugfix 处理，而不是只补文案。

### 2.3 单冲突首开定位

结论：

1. “只有 1 处冲突时，首开定位到该唯一冲突块”仍然是当前 PR 应有的既定行为。
2. 但 owner 最新现场观察已明确指出：第一次打开三栏合并窗口时看不到冲突点，必须手动点击上一个/下一个后才看到。
3. 因此，本轮 requirement truth 必须把这条收口为首开定位缺陷，不再沿用“仅需文档化预期”的旧结论。

### 2.4 为什么不是新增 UX 要求

本轮不是 owner 新发明一个更高阶的 merge 能力，而是对当前 PR 已经声称存在的行为给出了现场反例：

1. 当前冲突高亮是否本就应该可见。
2. 单冲突首开是否本就应该定位。

因此本轮不是新增 UX，而是“既定行为与 owner 现场观察不一致”的缺陷收口。

## 2.5 问题定义

问题定义统一收口为：

1. 用户第一次打开三栏 merge 工作台时，看不到当前冲突点。
2. 用户必须手动点击 `上一处冲突` 或 `下一处冲突` 后，当前冲突点才变得可见。
3. 该问题发生在“首帧定位 / 首帧可见性”层，不要求先区分是纯定位失败、纯高亮未出，还是首帧渲染时机问题。
4. 对产品与验收而言，只要 owner 观察到“第一次打开看不到，点导航后才看到”，就按缺陷处理。

## 3. 最小复现步骤

1. 准备 1 个 external edit 冲突文件，并确保最终只生成 1 个 merge conflict block。
2. 从 File Manager 进入统一 pending / conflict 入口。
3. 点击 `合并`，打开三栏 merge 工作台。
4. 第一次打开后，不做任何手动导航操作，直接观察工作台首帧状态。
5. 若首帧看不到冲突点，再手动点击 `上一处冲突` 或 `下一处冲突`，观察冲突点是否随后出现。

最小观察点：

1. 首帧是否能直接看到当前冲突点
2. 若是单冲突场景，计数器是否显示 `1 / 1`
3. 若是单冲突场景，`上一处冲突` 与 `下一处冲突` 是否 disabled
4. 手动点击导航后，冲突点是否才变得可见

## 4. 期望行为

### 4.1 首开期望

用户点击 `合并` 打开三栏 merge 工作台后，系统首帧就应让用户知道“当前正在处理哪一处冲突”，而不是第一次打开看不到冲突点、需要手动点导航后才知道。

### 4.2 当前冲突高亮期望

1. 当前冲突块在首帧即应可见。
2. 当前冲突块应相对普通冲突块有更强的 line / gutter emphasis。
3. 用户不应需要先点击 `上一处` / `下一处` 才知道当前焦点块在哪里。

### 4.3 单冲突定位期望

1. 若当前只有 1 处冲突，则工作台首开即默认定位到该唯一冲突块。
2. 计数器应显示 `1 / 1`。
3. `上一处冲突` 与 `下一处冲突` 应为 disabled。
4. 单冲突场景即便导航 disabled，也不能出现“首帧看不到冲突点”的结果。

## 5. 验收口径

### 5.1 PASS 口径

以下现象同时满足时，视为符合本轮 requirement：

1. 三栏 merge 第一次打开时，用户无需手动点导航，就能直接看到当前冲突点
2. 若是单冲突场景，计数器显示 `1 / 1`
3. 若是单冲突场景，前后导航按钮为 disabled
4. 当前冲突块在首帧已有专属高亮，而不是普通块同色无差别显示

### 5.2 FAIL 口径

出现以下任一现象，直接判 FAIL：

1. 第一次打开三栏 merge 工作台时，看不到当前冲突点
2. 必须点击 `上一处` / `下一处` 之后，当前冲突点或当前块高亮才出现
3. 单冲突场景首开未定位到唯一冲突块
4. 单冲突场景下，虽然显示 `1 / 1` 或导航 disabled，但首帧仍看不到冲突点
5. 单冲突场景的计数或导航状态与 `1 / 1` 事实不一致

## 6. 受影响面

1. compare / merge 工作台的首开定位与首帧可见性缺陷定义
2. QA focused 验收口径
3. PR 描述中的 reviewer-facing 行为说明
4. 后续 SA / DEV 的 bugfix 输入边界

本轮不直接扩成新的实现方案，也不重开 multi-conflict 导航、三栏布局、Monaco 技术细节或更大范围的 compare/merge 能力讨论。

2. 不在本轮 requirement 包里判定具体技术根因
3. 不重开 compare 双栏、冲突动作矩阵、pending dialog 或恢复语义

## 8. 与 PR 描述的关系

### 8.1 是否需要补进 PR 描述

需要。

原因：

1. owner 已现场观察到“第一次打开看不到冲突点，手动点导航后才看到”，因此 PR 描述不能再把首开定位写成已自然成立的体验而不注明这是本轮需要修正的缺陷点
2. reviewer 需要知道这条是当前已收口的 bugfix 边界，而不是纯文案优化

### 8.2 需要补入的文案点

PR 描述至少应体现以下 3 点：

1. merge 工作台的目标行为是首开即看到当前冲突点，不需要手动点导航后才看到
2. 单冲突场景目标行为仍是默认定位唯一冲突块，并显示 `1 / 1`
3. 当前冲突块高亮属于首帧即可见的目标行为

### 8.3 文档分层要求

1. 本文档是本轮 canonical requirement truth
2. `docs/plans/2026-05-15-task-43-po-pr-description-polish.md` 只保留 PR copy 与对本文档的引用
3. 后续 reviewer / SA / PM 若讨论本轮边界，以本文档为准，不再把 PR 文案草稿当 requirement truth

## 8.4 对后续 SA 的影响

1. SA 必须把本轮视为明确 bugfix 输入，而不是继续按“未确认 bug”处理。
2. SA 方案需要覆盖“为什么第一次打开看不到、手动点导航后才看到”的修复边界，但无需在 PO 阶段预写技术实现。
3. SA 不得仅通过调整 PR 文案或测试描述关闭该问题。
